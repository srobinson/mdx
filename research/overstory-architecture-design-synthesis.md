## Summary

Overstory is a multi-agent orchestration system that turns a single Claude Code session into a coordinated swarm of AI coding agents. It spawns workers in git worktrees via tmux, coordinates them through a purpose-built SQLite mail system, and merges completed branches with tiered conflict resolution. The core philosophical stance is: the orchestrator IS a Claude Code session — no separate daemon, no server process. Claude Code + hooks + the `ov` CLI is the whole system.

---

## Design Decisions and Why

### 1. The Orchestrator IS the Session

The user's Claude Code session IS the orchestrator. `SessionStart` and `UserPromptSubmit` hooks (`ov prime`, `ov mail check --inject`) bootstrap context at startup and surface incoming agent messages before each prompt. This eliminates a class of coordination overhead: there is no separate orchestration process to keep alive, restart, or monitor. The user's own context is the coordination brain.

### 2. SQLite Mail Over Everything Else

Agent communication uses a purpose-built SQLite mail store (WAL mode, ~1-5ms per query) rather than filesystem polling, sockets, or shared memory. The rationale stated in CLAUDE.md: beads (the task tracker) is too slow for high-frequency polling. SQLite WAL enables multiple concurrent writers (multiple agents) without contention. The mail schema includes typed messages (`status`, `question`, `result`, `error`, `worker_done`, `merge_ready`, `merge_failed`, `dispatch`, `escalation`, `health_check`), priorities, and threading.

### 3. Two-Layer Agent Instructions (Base + Overlay)

Every agent gets exactly two instruction layers:

- **Base definition** (files in `agents/*.md`): defines HOW — workflow, constraints, failure modes, communication protocol. These are reusable across any task.
- **Per-task overlay** (generated by `ov sling`, written to the worktree's instruction path): defines WHAT — task ID, file scope, spec path, branch name, parent agent.

The orchestrator or team lead only specifies WHAT. HOW is already embedded. This separation keeps per-task payloads small and keeps agent behavior consistent.

### 4. Git Worktrees as Isolation Units

Each agent gets its own git worktree — a lightweight linked checkout of the repo at a different path. This gives each agent a separate branch, a separate filesystem view, and physical isolation from other agents' writes. Agents commit only to their worktree branch; merging is handled by the coordinator or a dedicated merger agent. This prevents the most common multi-agent failure: two agents clobbering each other's changes.

### 5. Non-Overlapping File Scope as a Hard Invariant

The system treats overlapping file ownership as a system-level bug, not a recoverable error. The lead agent's instructions explicitly name `OVERLAPPING_FILE_SCOPE` as a named failure mode. Every file must have exactly one owner. The coordinator's instructions prohibit overlapping file areas across leads for the same reason. This trades flexibility for merge predictability.

### 6. Typed Failure Modes Over Generic Error Handling

Every agent definition enumerates specific named failures (`READ_ONLY_VIOLATION`, `SPEC_WITHOUT_SCOUT`, `HIERARCHY_BYPASS`, `PREMATURE_MERGE`, etc.). These are not just documentation — they are behavioral checkpoints embedded in the agent's instructions. The agent is told: "if you catch yourself doing any of these, stop and correct immediately." This encodes error recovery into the agent's operating mode rather than relying on external detection.

### 7. Strict Hierarchy Enforcement

The coordinator may only spawn leads. Leads may only spawn scouts, builders, reviewers, and mergers. This is enforced at the CLI level (`sling.ts` throws `HierarchyError` on violations). The depth limit (default 2) is configurable. The rationale: unbounded recursive delegation produces exponentially growing agent counts. The 3-tier hierarchy (coordinator → lead → workers) gives 4-25 effective workers per coordinator while keeping the tree shallow enough to monitor.

### 8. Merge Authorization Through Explicit Signals

The coordinator does not merge branches based on `ov status` showing agents completed, watchdog nudges, or its own git inspection. Only a typed `merge_ready` mail from the owning lead authorizes a merge. `PREMATURE_MERGE` is a named failure mode for the coordinator. This preserves the invariant that leads own quality verification — only after a lead has run quality gates and optionally spawned a reviewer does the `merge_ready` signal go out.

### 9. Minimal Runtime Dependencies

The package has three runtime dependencies: `chalk` (color output), `commander` (CLI framework), and `@os-eco/mulch-cli` (expertise API). All I/O uses Bun built-in APIs (`bun:sqlite`, `Bun.spawn`, `Bun.file`). External tools (`git`, `tmux`, `bd`/`sd`, `mulch`) are called as subprocesses. This keeps the install footprint minimal and avoids the dependency graph problems that plague Node.js CLIs.

## Agent Role System

The system defines 7 agent roles, arranged in a strict hierarchy.

### Coordinator (depth 0)

The strategic center. Receives high-level objectives from the human, decomposes into work streams, creates tracker issues, dispatches leads, monitors via mail and `ov status`, handles escalations by severity, and merges completed branches after explicit `merge_ready` signals.

**Hard constraints:** No Write or Edit access to any file. No spec writing. Can only spawn leads. Runs at project root, not in a worktree. All state is in external systems (tracker, mail.db, sessions.db) — not in its conversation history.

**Operator interaction:** When mail arrives from the `operator` sender (the human via `ov coordinator ask`), the coordinator treats it as a synchronous request and must reply with a structured status format. `correlationId` must be echoed back in the payload.

**Exit triggers:** Configurable conditions (`allAgentsDone`, `taskTrackerEmpty`, `onShutdownSignal`) that cause the coordinator to gracefully stop. When all triggers are met, it syncs state, marks the run complete, and stops.

### Lead (depth 1)

The team coordinator. Receives a high-level work stream, decomposes it into subtasks, spawns scouts/builders/reviewers, monitors their progress, and signals `merge_ready` to the coordinator when each builder's branch passes verification.

**Complexity-adaptive workflow:** The lead uses a three-tier complexity assessment:
- Simple tasks (1-3 files, well-understood): lead does the work directly
- Moderate tasks (3-6 files, focused scope): spawn one builder, lead self-verifies by reading the diff
- Complex tasks (6+ files, multiple subsystems): full Scout → Build → Verify pipeline

**Spec ownership:** The coordinator writes no specs. The lead owns spec production — scouts explore, leads synthesize findings into grounded specs, builders implement against those specs. A spec without prior exploration is a named failure (`SPEC_WITHOUT_SCOUT`).

**File scope partitioning:** The lead ensures non-overlapping file ownership. Each builder gets a disjoint file set. "Every file belongs to exactly one builder" is the invariant.

### Scout (depth 2, leaf)

Read-only exploration. Investigates codebases, traces patterns, follows imports, checks docs and configs, and reports findings. The one allowed write operation is `ov spec write` for persisting spec files.

**Communication pattern:** Scouts send a short result mail (1-2 sentences) pointing to the spec file path. The spec file contains the details. This keeps mail costs low.

**Classification guidance:** When reporting notable findings, scouts indicate suggested classification (`foundational`, `tactical`, `observational`) to help the lead record accurately via mulch.

### Builder (depth 2, leaf)

Implementation specialist. Reads its overlay and spec, loads expertise, implements within its FILE_SCOPE, runs quality gates, commits to its worktree branch, records mulch learnings, sends `worker_done` to parent, closes the tracker issue, and exits.

**Mulch as a required gate:** Recording learnings via `ml record` is explicitly required before closing. `MISSING_MULCH_RECORD` is a named failure mode. The rationale: every implementation session produces reusable insights (conventions discovered, patterns applied, failures encountered). Capturing them prevents future agents from repeating the same mistakes.

### Reviewer (depth 2, leaf)

Read-only validation. Reviews code changes via `git diff`, checks correctness, style, security, test coverage, runs quality gates, and reports PASS or FAIL with actionable feedback. Sends detailed review mail to the lead.

### Merger (depth 2, leaf)

Branch integration specialist. Follows a four-tier conflict resolution escalation:

1. **Tier 1 (Clean):** `git merge --no-edit`. If exit code 0, done.
2. **Tier 2 (Auto-resolve):** Parse conflict markers, resolve simple non-overlapping changes automatically.
3. **Tier 3 (AI-resolve):** Read both versions, understand intent from specs and commit messages, produce a merged version that preserves both intents.
4. **Tier 4 (Reimagine):** Last resort. Start from clean checkout, reimplement changes from scratch against current target state.

`TIER_SKIP` (jumping tiers without attempting lower ones first) is a named failure mode.

### Monitor (persistent, no worktree)

Continuous fleet patrol agent. Runs at the project root without a worktree. Monitors agent health, detects zombies, surfaces stalls. A Tier 2 watchdog complement to the Tier 0 mechanical watchdog daemon.

## Runtime Abstraction Layer

The original system was tightly coupled to Claude Code: binary name, CLI flags, TUI readiness strings, hook system, `.claude/` conventions, and transcript format were hardcoded across 35 coupling points in 15+ files.

### The Problem Statement

A Claude Code-only system locks the entire swarm to a single runtime. The goal: a thin interface layer where each new runtime is a ~200-400 line adapter file, not a fork of the orchestration engine.

### The AgentRuntime Interface

Six methods on the interface, two optional:

```typescript
interface AgentRuntime {
  id: string;
  readonly instructionPath: string;

  buildSpawnCommand(opts: SpawnOpts): string;       // tmux session command
  buildPrintCommand(prompt: string, model?: string): string[];  // headless one-shot call
  deployConfig(worktreePath, overlay, hooks): Promise<void>;    // write instructions + guards
  detectReady(paneContent: string): ReadyState;     // TUI readiness from tmux capture
  parseTranscript(path: string): Promise<TranscriptSummary | null>;  // normalize token usage
  buildEnv(model: ResolvedModel): Record<string, string>;       // provider env vars

  requiresBeaconVerification?(): boolean;   // whether to resend startup prompt
  connect?(process: RpcProcessHandle): RuntimeConnection;  // optional RPC bypass for tmux
}
```

The `instructionPath` property captures the most fundamental difference between runtimes: where the agent reads its instructions from.

| Runtime | instructionPath |
|---------|----------------|
| Claude Code | `.claude/CLAUDE.md` |
| Codex | `AGENTS.md` |
| Pi | `.claude/CLAUDE.md` (reads natively) |
| Copilot | `.github/copilot-instructions.md` |
| Cursor | `.cursor/rules/overstory.md` |

### The Five Built-in Adapters

**Claude Code** (`claude.ts`): Full TUI lifecycle via tmux. Two-phase readiness detection (trust dialog → ready prompt). PreToolUse hooks in `.claude/settings.local.json` provide security guards. Beacon verification required (Claude Code's TUI can swallow Enter during startup). Transcript parsing delegates to `src/metrics/transcript.ts`.

**Codex** (`codex.ts`): Headless runtime. `codex exec --full-auto --json` processes a task and exits. Always returns `{ phase: "ready" }`. No hook deployment — OS-level sandbox (Seatbelt on macOS, Landlock on Linux) via `--full-auto` replaces hook-based guards. NDJSON stdout (`--json`) replaces hook-based event logging. No beacon verification. Token usage from `turn.completed` events.

**Pi** (`pi.ts`): TUI runtime (Mario Zechner's Pi coding agent). Reads `.claude/CLAUDE.md` natively — zero overlay translation needed. Security via TypeScript extension (`pi-guards.ts` generates `.pi/extensions/overstory-guard.ts`). No beacon verification — Pi's idle and processing states are visually indistinguishable, so the resend loop would cause duplicate prompts. RPC mode (`connect()`) not yet implemented but designed for.

**Copilot** (`copilot.ts`): GitHub Copilot CLI. `--allow-all-tools` maps to bypass mode. No hook mechanism — `_hooks` parameter unused. `appendSystemPrompt` silently ignored (no CLI flag equivalent). Dual-format transcript parser (handles both Claude-style and Pi-style token events).

**Cursor** (`cursor.ts`): Cursor CLI (`agent` binary). `--yolo` maps to bypass mode. Instruction file is `.cursor/rules/overstory.md`. Token usage not available from stream-json format (returns zeros). No per-tool interception hooks.

### Guard System Design

Guards enforce security boundaries. The mechanism differs by runtime but the shared constants live in `src/agents/guard-rules.ts`:

- `NATIVE_TEAM_TOOLS`: Claude Code native delegation tools — agents must use `ov sling` instead
- `INTERACTIVE_TOOLS`: Tools requiring human interaction — agents must escalate via mail
- `WRITE_TOOLS`: Blocked for non-implementation capabilities (scouts, reviewers, coordinator)
- `DANGEROUS_BASH_PATTERNS`: `git push`, `git reset --hard`, wrong branch names
- `SAFE_BASH_PREFIXES`: Whitelist checked before the blocklist

All Claude Code hooks include an `ENV_GUARD` prefix that makes guards no-ops for the user's own Claude Code session at the project root (checked via `$OVERSTORY_AGENT_NAME`).

### TUI vs Headless Lifecycle

**TUI runtimes** (Claude Code, Pi, Copilot): persistent tmux pane, readiness detection loop, send-keys for mail delivery, SIGTERM for shutdown.

**Headless runtimes** (Codex): spawn process, task processes and exits, NDJSON events on stdout, `proc.exited` for completion detection. No readiness detection, no beacon verification, no mid-execution mail delivery.

**Mixed swarms:** A single swarm can mix runtimes per-agent. `ov sling task-001 --runtime claude` and `ov sling task-002 --runtime codex` in the same session. The registry resolves the appropriate adapter per agent.

### The RPC Path (Future)

The `connect?()` optional method on the interface defines a `RuntimeConnection` interface for runtimes that support direct stdin/stdout JSON-RPC. When implemented, the orchestrator uses `connection.followUp()` instead of tmux send-keys for mail delivery and `connection.abort()` instead of SIGTERM. Pi is designed for this but `connect()` is not yet wired. This path eliminates tmux as a coupling point entirely for capable runtimes.

### Implementation Phases (Design Document)

- **Phase 0**: Extract the interface. Zero behavior change. Claude behavior moves behind the interface.
- **Phase 1**: Abstract the headless print command used by merge resolver and watchdog triage.
- **Phase 2**: Codex adapter. NDJSON stdout capture, AGENTS.md instruction delivery.
- **Phase 3**: Pi adapter. RPC client, guard extension deployment.
- **Phase 4**: Generic ACP adapter (when Agent Client Protocol stabilizes across OpenCode, Cline, etc.).

## Philosophical Approach

### 1. The Session IS the Coordinator

Multi-agent orchestration does not require a separate daemon. The user's existing Claude Code session — with appropriate hooks — is a fully capable orchestration brain. This keeps the system from requiring infrastructure that must be operated and maintained independently.

### 2. Propulsion Principle

Every agent definition opens with the same directive: "Read your assignment. Execute immediately. Do not ask for confirmation, do not propose a plan and wait for approval, do not summarize back what you were told." This is the "propulsion principle" — reduce turn-around latency by eliminating conversational overhead. Agents are designed to start acting within their first tool call.

### 3. Cost Awareness as First-Class Concern

Every agent definition includes a cost-awareness section. For the lead: "Your time is the scarcest resource in the swarm. As the lead, you are the bottleneck — every minute you spend reading code is a minute your team is idle." For builders: "Every mail message and every tool call costs tokens." Scouts and reviewers are framed as quality investments, not overhead. The lead is explicitly told where to actually save tokens (fewer, well-scoped builders; batched status updates).

### 4. Named Failures Over Generic Error Recovery

Rather than generic "handle errors" instructions, each agent role has a curated list of named failure modes with explicit self-correction instructions. This gives the LLM concrete vocabulary for recognizing and correcting its own behavior patterns. `SPEC_WITHOUT_SCOUT`, `PREMATURE_MERGE`, `HIERARCHY_BYPASS`, `OVERLAPPING_FILE_SCOPE` are all failure labels the agent can match against its own actions in real time.

### 5. External State Over Conversation History

The coordinator is long-lived and survives compaction, restarts, and session handoffs. This works because all meaningful state is in external systems: tracker issues, SQLite mail, sessions.db, groups.json, checkpoint.json. The coordinator's context recovery protocol reads these external systems on startup rather than depending on conversation history. The system is designed to be crash-tolerant.

### 6. Extract, Don't Rewrite

The runtime abstraction design principle is "Extract, don't rewrite." Phase 0 moves current Claude Code behavior behind an interface with zero behavior change. New runtimes come after. This is the minimum viable abstraction path — get the seam right first, then add implementations.

### 7. Real Implementations Over Mocks

The testing philosophy explicitly states: "Never mock what you can use for real." Real filesystem (temp dirs), real SQLite (`:memory:`), real git repos. Only mock when the real thing has unacceptable side effects: tmux (interferes with developer sessions), external AI services (cost and latency), network requests (flaky in CI).
