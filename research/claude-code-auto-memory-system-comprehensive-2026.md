## Executive Summary

Claude Code has three distinct memory systems operating at different layers: (1) **CLAUDE.md files** (user-written instructions), (2) **auto-memory** (Claude-written learnings stored at `~/.claude/projects/<project>/memory/`), and (3) the **API-level Memory Tool** (`memory_20250818`), a client-side tool for the Messages API. The subagent memory system adds a fourth layer with three scopes (`user`, `project`, `local`). The system prompt that defines four memory types (user, feedback, project, reference) with `when_to_save`/`how_to_use` fields is generated by Claude Code at runtime and injected when the `memory` frontmatter field is set in a subagent definition. The Piebald-AI/claude-code-system-prompts repository has reverse-engineered nearly all prompt fragments.

## 1. Auto-Memory (Main Session)

### What it is
Auto-memory lets Claude accumulate knowledge across sessions without the user writing anything. Claude saves notes for itself as it works: build commands, debugging insights, architecture notes, code style preferences, and workflow habits. Claude decides what to remember based on whether the information would be useful in a future conversation.

**Requires Claude Code v2.1.59 or later.** Enabled by default.

### Storage location
Each project gets its own memory directory:
```
~/.claude/projects/<project>/memory/
  MEMORY.md          # Concise index, loaded into every session
  debugging.md       # Topic files Claude creates as needed
  api-conventions.md
  ...
```

The `<project>` path is derived from the **git repository root**. All worktrees and subdirectories within the same repo share one auto-memory directory. Outside a git repo, the project root is used instead.

### Path encoding
Paths are encoded by replacing `/` with `-`. Examples from a real system:
- `/Users/alphab/Dev/LLM/DEV/helioy/fmm/` becomes `-Users-alphab-Dev-LLM-DEV-helioy-fmm`
- `/Users/alphab/.claude/` becomes `-Users-alphab--claude` (double dash because `.claude` starts with `.`)
- `/private/tmp/` becomes `-private-tmp`

On macOS, symlinked `/tmp` resolves to `/private/var/folders/...`, producing long encoded paths.

### How MEMORY.md is loaded
1. First 200 lines of `MEMORY.md` are loaded at the start of every conversation
2. Content beyond line 200 is NOT loaded
3. Topic files (e.g., `debugging.md`) are NOT loaded at startup; Claude reads them on demand
4. The 200-line limit applies ONLY to `MEMORY.md`, not to `CLAUDE.md` files

**Internal implementation** (from bundle analysis by Giuseppe Gurgone):
- Memory file identifier: `U_ = "MEMORY.md"`
- Line limit constant: `pZ = 200`
- These are literal values found in Claude Code's minified CLI bundle

### Feature flag
`tengu_coral_fern` (disabled by default) controls automatic reading of topic files referenced in MEMORY.md. When enabled, Claude would automatically load relevant topic files without needing explicit tool calls.

### Configuration
```json
// .claude/settings.json or settings.local.json
{
  "autoMemoryEnabled": false,          // disable auto-memory
  "autoMemoryDirectory": "~/custom/"   // redirect storage location
}
```

Environment variable: `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`

The `autoMemoryDirectory` setting is accepted from policy, local, and user settings but NOT from project settings (`.claude/settings.json`) to prevent a shared project from redirecting auto-memory writes to sensitive locations.

### Empty state prompt
When MEMORY.md is empty, this message is shown to Claude:
> "Your MEMORY.md is currently empty. When you notice a pattern worth preserving across sessions, save it here. Anything in MEMORY.md will be included in your system prompt next time."

### Known bug (issue #24474)
Claude sometimes creates separate `.md` files in the memory directory that are never read back because only `MEMORY.md` is auto-loaded. These "orphaned" files accumulate without being referenced. The issue was labeled `bug`, `has repro`, `memory` and closed as stale.

## 2. Subagent Memory (`memory` frontmatter field)

### Frontmatter syntax
Add `memory` to the YAML frontmatter of a subagent definition file:

```yaml
---
name: code-reviewer
description: Reviews code for quality and best practices
tools: Read, Write, Edit, Bash
model: sonnet
memory: user
---

You are a code reviewer...
```

### Three memory scopes

| Scope     | Storage Location                              | Version Controlled | Shared | Use Case |
|-----------|-----------------------------------------------|-------------------|--------|----------|
| `user`    | `~/.claude/agent-memory/<agent-name>/`        | No                | No     | Cross-project knowledge |
| `project` | `.claude/agent-memory/<agent-name>/`          | Yes               | Yes    | Project-specific, team-shared |
| `local`   | `.claude/agent-memory-local/<agent-name>/`    | No (gitignored)   | No     | Personal project knowledge |

`project` is the recommended default in official docs. `user` is recommended for broadly applicable knowledge.

### What happens when memory is enabled

1. The subagent's system prompt includes instructions for reading and writing to the memory directory
2. The first 200 lines of `MEMORY.md` in the memory directory are injected into the system prompt
3. Read, Write, and Edit tools are automatically enabled (even if not listed in `tools`) so the subagent can manage its memory files
4. Instructions include the four memory types taxonomy (user, feedback, project, reference) with `when_to_save`, `how_to_use`, and `body_structure` guidance

### CLI-defined agents
The `memory` field also works with `--agents` JSON:
```bash
claude --agents '{"reviewer": {"description": "...", "memory": "user"}}'
```

### Agent memory directory naming
The `<agent-name>` in the path comes from the `name` field in frontmatter. For plugin-provided agents, the directory name is prefixed with the plugin name (e.g., `helioy-tools-deep-research`).

## 3. The Memory System Prompt (Four Types)

The large block of instructions about memory types (user, feedback, project, reference) that gets injected into subagent system prompts is **generated by Claude Code at runtime** when the `memory` frontmatter field is set. The template is built into Claude Code's codebase, not user-configurable.

### Evidence from Piebald-AI reverse engineering

The Piebald-AI/claude-code-system-prompts repository has captured individual fragments of this prompt across multiple versions:

**`system-prompt-description-part-of-memory-instructions.md`** (since v2.1.69):
Describes the "user" memory type: information about the user's role, goals, responsibilities, and knowledge.

**`system-prompt-memory-description-of-user-feedback.md`** (since v2.1.78):
Describes the "feedback" memory type: guidance about work approaches, recording both successes and failures, checking for contradictions with team memories.

**`system-prompt-agent-memory-instructions.md`** (since v2.1.31):
Instructions for including domain-specific memory update guidance in agent system prompts. Provides template patterns like:
> "Update your agent memory as you discover [domain-specific items]. This builds up institutional knowledge across conversations."

**CHANGELOG v2.1.71**: First appearance of "Memory system (private feedback)" prompt
**CHANGELOG v2.1.78**: Replaced with "Memory description of user feedback" (current form)

### The four memory types

These are the types defined in the injected system prompt:

| Type | Purpose | When to Save | Structure |
|------|---------|-------------|-----------|
| `user` | User's role, goals, knowledge, preferences | When you learn details about user's role, preferences, responsibilities | Free-form |
| `feedback` | Guidance on approach (corrections AND confirmations) | Any correction OR confirmation of non-obvious approach | Rule, then **Why:** and **How to apply:** lines |
| `project` | Ongoing work, goals, bugs, incidents | When you learn who is doing what, why, or by when | Fact/decision, then **Why:** and **How to apply:** |
| `reference` | Pointers to external systems | When you learn about resources in external systems | Pointer with purpose |

### Memory file format
Individual memory files use this frontmatter:
```markdown
---
name: {{memory name}}
description: {{one-line description}}
type: {{user, feedback, project, reference}}
---

{{memory content}}
```

MEMORY.md is an index with pointers to these files, not a memory dump. The distinction: MEMORY.md links to files with brief descriptions; memory content lives in the individual files.

### What NOT to save (from the injected prompt)
- Code patterns, conventions, architecture, file paths (derivable from code)
- Git history, recent changes (git log/blame are authoritative)
- Debugging solutions (fix is in the code; commit message has context)
- Anything in CLAUDE.md files
- Ephemeral task details

## 4. Memory Selection Agent

Since v2.1.74, Claude Code uses a separate lightweight agent to determine which memory files to attach for each query.

**`agent-prompt-determine-which-memory-files-to-attach.md`** (v2.1.75):
> "You are selecting memories that will be useful to Claude Code as it processes a user's query. Return a list of filenames (up to 5). Only include memories that you are certain will be helpful. If a list of recently-used tools is provided, do not select memories that are usage reference or API documentation for those tools -- DO still select memories containing warnings, gotchas, or known issues."

This replaced a simpler "Memory selection" agent in v2.1.74.

## 5. Dream Memory Consolidation

Since v2.1.78, Claude Code has a "dream" mechanism: a reflective pass over memory files.

**`agent-prompt-dream-memory-consolidation.md`**:
Four phases:
1. **Orient**: Read existing memories and MEMORY.md index
2. **Gather recent signal**: Search daily logs, existing memories, JSONL transcripts (grep narrowly)
3. **Consolidate**: Merge new signal into existing topic files; convert relative dates to absolute; delete contradicted facts
4. **Prune and index**: Keep MEMORY.md index under 200 lines; remove stale pointers; resolve contradictions

## 6. Session Memory (Separate System)

Session memory is distinct from auto-memory. It uses `summary.md` files with a fixed template structure:

```
# Session Title
# Current State
# Task specification
# Files and Functions
# Workflow
# Errors & Corrections
# Codebase and System Documentation
# Learnings
# Key results
# Worklog
```

Updated via `agent-prompt-session-memory-update-instructions.md` (since v2.0.58). The update agent can ONLY use Edit tool calls and must preserve the template structure (headers and italic descriptions) exactly.

## 7. API-Level Memory Tool

The `memory_20250818` tool is a completely separate system from Claude Code's auto-memory. It is a client-side tool for the Messages API that provides file-system-like operations (view, create, str_replace, insert, delete, rename) against a `/memories` directory.

Key differences from Claude Code auto-memory:
- API-level, not CLI-level
- Client implements the storage backend (file, database, cloud, encrypted)
- No `MEMORY.md` index pattern
- No memory types taxonomy
- Works via tool_use/tool_result message pairs
- Composable with Context Editing and Compaction

---

## 8. Token Overhead Concern (Issue #6825)

Subagents inherit system prompt, user memory, and project memory by default. In one test, a "tiny" subagent running `echo success` consumed ~60k tokens because it inherited full parent memory and context. Without memory, the same subagent consumed ~3k tokens. This led to a request for configurable memory inheritance, which the `memory` frontmatter field partially addresses.

---

## 9. Team Memory

Since v2.1.71, there is a "team memory" concept with shared content:

**`system-prompt-team-memory-content-display.md`**:
```
Contents of ${MEMORY_ITEM.path}${MEMORY_TYPE_DESCRIPTION}:

<team-memory-content source="shared">
${MEMORY_CONTENT}
</team-memory-content>
```

Team memories are wrapped in `<team-memory-content source="shared">` tags. The feedback type specifically instructs checking for contradictions against team feedback before saving private feedback.

## Open Questions

1. **How does the memory system prompt assemble the four types?** Individual fragments exist in the Piebald-AI repo, but the full assembled template has not been captured as a single file. It appears to be constructed programmatically from multiple fragments.

2. **What triggers auto-memory writes?** The official docs say Claude "decides what's worth remembering." The mechanism for this decision is internal to the model's behavior shaping, not a deterministic code path.

3. **How does memory selection interact with the memory types?** The selection agent receives filenames and descriptions but it's unclear whether it uses the `type` frontmatter to prioritize.

4. **Will `tengu_coral_fern` be enabled by default?** This would make topic files auto-loadable, significantly changing memory utility.

5. **How does git repo detection work across worktrees?** The docs say "all worktrees share one auto memory directory" but the path encoding mechanism for worktrees versus subdirectories within a repo is not documented.

## Actionable Takeaways

1. **The `memory: user` scope in your agent definitions** stores memory at `~/.claude/agent-memory/<plugin-name>:<agent-name>/` (or just `<agent-name>/` for non-plugin agents). This is exactly what you see in `~/.claude/agent-memory/helioy-tools-deep-research/`.

2. **MEMORY.md is an index.** Keep it under 200 lines. Put detailed knowledge in topic files. The system only auto-loads MEMORY.md; topic files are read on demand.

3. **The four memory types** (user, feedback, project, reference) with their `when_to_save`/`how_to_use`/`body_structure` fields are part of Claude Code's built-in system prompt injection. They are not user-configurable. The template is assembled from fragments in the codebase.

4. **Dream consolidation** is the mechanism for keeping memory files clean over time. It reads transcripts, merges signal into topic files, converts relative dates to absolute, and prunes the index.

5. **The API-level Memory Tool** (`memory_20250818`) is a completely different system. It provides file-system-like operations via tool_use calls and requires client-side implementation. Do not conflate it with Claude Code's auto-memory.
